## 1. 命名规范

### 1.1 文件命名

| 页面类型 | 命名格式 | 示例 |
|---------|---------|------|
| 源笔记 | `src_YYYYMMDD_NNN` | `src_20260530_001` |
| 概念 | `cpt_[canonical_name]` | `cpt_attention_mechanism` |
| 框架 | `frm_[name]` | `frm_chain_of_thought` |
| Claim | `clm_[short_desc]` | `clm_scaling_law_emergence` |
| 问答 | `qa_YYYYMMDD_NNN` | `qa_20260530_001` |
| 比较 | `cmp_[items]` | `cmp_transformer_vs_mamba` |

**规则**：
- 全小写，单词用 `_` 连接
- 英文命名，中文标题放在 frontmatter `title` 字段
- 文件名保持稳定，不因内容修改而改名

### 1.2 Frontmatter 规范

```yaml
---
id: src_20260530_001          # 稳定 ID
title: 页面标题                # 中文标题
created: 2026-05-30            # 创建日期
updated: 2026-05-30            # 最后更新日期
type: source-note              # source-note / concept / framework / claim / qa / comparison
tags: [tag1, tag2]             # 标签（从标签分类中选取）
sources:                       # 来源引用
  - raw/papers/xxx.pdf
confidence: high               # high / medium / low / speculative
contested: false               # 是否存在争议
contradictions:                # 矛盾信息
  - src_20260520_003           # 矛盾来源 ID
---
```

### 1.3 Wikilinks 规范

```markdown
# 同目录链接
[[概念名称]]

# 带显示文本的链接
[[cpt_attention|注意力机制]]

# 跨目录链接
[[../concepts/概念名称]]
```

## 3. 页面创建阈值

### 3.1 创建新页面

| 条件 | 动作 |
|------|------|
| 概念在 2+ 来源中被提及 | ✅ 创建概念页 |
| 出现新的独立框架 | ✅ 创建框架页 |
| 出现明确的可验证 Claim | ✅ 创建 Claim 页 |

### 3.2 更新现有页面

| 条件 | 动作 |
|------|------|
| 新资料提及已有概念 | ✅ 补充到现有页面 |
| 新资料提供新角度 | ✅ 添加新 section |
| 新资料修正已有信息 | ✅ 更新 + 记录变更 |

### 3.3 不创建页面

| 条件 | 动作 |
|------|------|
| 仅在上下文中顺带提及 | ❌ 不创建，仅在源笔记中记录 |
| 信息过于笼统 | ❌ 不创建 |
| 无法独立成页的碎片 | ❌ 不创建 |

### 3.4 页面拆分

- 单页超过 **200 行**时，考虑拆分
- 拆分方式：按子主题拆分为独立概念页，原页改为索引/总览页

### 3.5 归档规则

- 被新资料明确否定的 Claim → 标记 `confidence: refuted`，保留在 wiki
- 被更好框架替代的旧框架 → 归档到 `archive/`，原位置添加重定向

## 4. 知识等级

### Level 0: 仅归档

```
处理方式：保存到 raw/，记录在 source-registry
不创建 wiki 页面
适用：与领域关联度低、或纯事实性记录
```

### Level 1: 轻量卡片

```
处理方式：创建源笔记（精简版），仅记录
- 来源信息
- 一句话摘要
- 关键 tag
适用：有价值但不需深度整理的资料
```

### Level 2: 标准编译

```
处理方式：完整源笔记 + 概念提取 + Claim 记录
- 完整 12 节源笔记
- 提取相关概念（创建或更新概念页）
- 记录核心 Claim
适用：高质量、与领域高度相关的资料
```

### Level 3: 系统级整合

```
处理方式：Level 2 + 框架识别 + 交叉引用 + 知识图谱更新
- 识别并记录框架
- 建立完整的交叉引用网络
- 更新 index.md 知识地图
- 生成可能的 Marp 幻灯片
适用：核心领域的重要资料
```

## 5. 变更类型 (Mutation Types)

| 类型 | 代码 | 说明 |
|------|------|------|
| 仅归档 | `archive_only` | 保存原始资料，不创建 wiki 页面 |
| 新知识 | `new_knowledge` | 创建新的 wiki 页面 |
| 补充 | `supplement` | 向现有页面添加新信息 |
| 修正 | `correction` | 修正现有页面的错误信息 |
| 证伪 | `falsification` | 标记现有 Claim 为已证伪 |
| 拆分 | `split` | 将一个页面拆分为多个 |
| 合并 | `merge` | 将多个页面合并为一个 |
| 待审核 | `pending_review` | 需要人类审核的变更 |

## 6. 人工审核门控 (Human Gate)

### 需要人工审核的操作

以下操作 **必须** 经过人工审核后才能执行：

| 操作 | 风险等级 | 说明 |
|------|---------|------|
| 修正高置信度 Claim | 🔴 高 | 修改 `confidence: high` 的已有 Claim |
| 证伪操作 | 🔴 高 | 标记已有知识为已证伪 |
| 合并概念 | 🟡 中 | 将多个概念合并为一个 |
| 拆分概念 | 🟡 中 | 将一个概念拆分为多个 |
| 重命名/移动页面 | 🟡 中 | 改变页面 ID 或位置 |
| 修改框架 | 🟡 中 | 对已有框架进行结构性修改 |

### 无需审核的操作

| 操作 | 说明 |
|------|------|
| 新建页面 | 创建全新的 wiki 页面 |
| 补充信息 | 向现有页面添加新内容（不修改已有内容） |
| 更新索引 | 更新 index.md |
| 记录日志 | 追加 log.md |
| 创建源笔记 | 对新资料创建源笔记 |

### 审核流程

1. LLM agent 识别到需审核的操作
2. 创建 `pending_review` 记录到 `templates/changelog/pending-review.md`
3. 通知用户（通过飞书消息）
4. 用户确认后执行，或用户拒绝则标记为 `rejected`

## 7. 变更日志规范

### 格式

```markdown
## [YYYY-MM-DD HH:MM] 操作类型 | 简述

- 变更内容 1
- 变更内容 2
- 影响范围: page_id_1, page_id_2
- 来源: src_YYYYMMDD_NNN
- mutation: supplement
- 审核状态: auto / pending / approved
```

### 规则

- 日志为 **append-only**，不修改已有记录
- 超过 1000 行时，归档到 `log-archive/`
- 每条记录必须包含：时间、操作类型、影响范围

---

## 8. 更新策略

### 增量更新

- 所有写入操作默认为 **增量模式**
- 覆盖已有内容需显式 `--force` 标记
- 新信息追加到页面末尾或相关 section

### 冲突处理

- 当新信息与已有信息矛盾时：
  1. 保留两者，标记 `contested: true`
  2. 记录矛盾来源
  3. 如需判断，提交人工审核

### 一致性保障

- 每次变更后自动 `git commit`
- `index.md` 与实际页面保持同步
- 定期运行 `lint` 检查一致性

### 版本控制

- Wiki 目录使用 git 管理
- 每次操作一个 commit
- Commit message 格式：`[wiki] 操作类型: 简述`
